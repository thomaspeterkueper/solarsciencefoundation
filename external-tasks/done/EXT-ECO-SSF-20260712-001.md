# EXT-ECO-SSF-20260712-001 — Retention-Löschjob

## Anlass

Die Datenschutz-Policy (KG, §7 via `EXT-ECO-KG-20260712-002`) legt fest: Konten
ohne Aktivität werden nach 6 Monaten gelöscht, Registrierungen mit unbestätigter
E-Mail nach 1 Monat. In SSFs Supabase existiert derzeit **kein** Löschjob — ohne
Umsetzung wäre die Zusage in der Datenschutzerklärung unzutreffend.

## Gewünschte Änderung

Einen periodisch laufenden Löschmechanismus einrichten (technische Wahl offen:
`pg_cron` + `SECURITY DEFINER`-Funktion, geplante Edge Function oder externer
Cron). Zwei Regeln:

1. **Inaktive Konten (> 6 Monate):** SSF `profiles` besitzt kein
   `last_seen_at`; als Inaktivitäts-Signal `auth.users.last_sign_in_at < now() - interval '6 months'`
   verwenden (optional zusätzlich jüngste `learning_progress.updated_at` /
   `exercise_attempts`-Aktivität). Konto über `auth.users` löschen; die
   `ON DELETE CASCADE`-Ketten auf `profiles(id)` (u. a. `learning_progress`,
   `exercise_attempts`, `unlocks`, `user_achievements`, `member_roles`,
   `supporter_records`) greifen dann mit. `project_access_audit` steht auf
   `ON DELETE SET NULL` — Verhalten prüfen.
2. **Unbestätigte E-Mail (> 1 Monat):** löschen, wo
   `auth.users.email_confirmed_at IS NULL AND auth.users.created_at < now() - interval '1 month'`.

## Begründung

Die Umsetzung liegt in SSFs eigenem Supabase-Projekt und damit in SSFs
Zuständigkeit. Der zentrale Datenschutztext (KG) ist Source of Truth für die
Aussage, die Ausführung nicht.

## Betroffene Repositories

- `solarsciencefoundation` (Umsetzung)
- `kueper-knowledge-graph` (Release-Gate: §7 darf erst `released`, wenn dieser Job live ist)

## Erwartetes Ergebnis

Der Job läuft periodisch und löscht beide Fälle nachweislich; Kaskaden greifen
korrekt (inkl. Verhalten von `project_access_audit`); die Umsetzung ist als
Migration dokumentiert. Danach ist das Ist-Verhalten deckungsgleich mit §7.

## Hinweise

Vor Produktivschaltung ein Trockendurchlauf mit Logging (betroffene Zeilenzahl),
um versehentliche Massenlöschung auszuschließen. DSGVO-Löschung in Backups/Storage
ist hier nicht adressiert und ggf. separat zu betrachten. Ersetzt keine Rechtsberatung.


## Resolution — 2026-07-15

**Implementiert als Vercel-Cron:**
- Route: `app/api/cron/retention/route.ts` v1.0.0
- Schedule: täglich `0 3 * * *` (03:00 UTC) via `vercel.json`
- Auth: `lib/cronSecret.ts` (Bearer-Token)
- Trockenmodus: `RETENTION_DRY_RUN=true` (Default)

**SSF-Besonderheit:** `project_access_audit` → `ON DELETE SET NULL` berücksichtigt.
Audit-Einträge bleiben erhalten, `user_id` wird NULL.

**Noch zu tun:**
- `CRON_SECRET` + `SUPABASE_SERVICE_ROLE_KEY` in Vercel SSF-Projekt setzen
- `RETENTION_DRY_RUN=true` setzen, ersten Lauf prüfen
- Danach `RETENTION_DRY_RUN=false` aktivieren

**KG-Gate:** §7 kann auf `released` wenn erster produktiver Lauf bestätigt.
